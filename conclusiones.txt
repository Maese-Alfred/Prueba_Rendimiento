PROBLEMAS IDENTIFICADOS
=======================

Problema 1: TASA DE ERROR ALTA (9.39%)
Severidad: CRITICA
Evidencia: 507 requests fallaron de 5,399 totales; 1,014 checks fallidos
Impacto: El servicio NO cumple con el SLA de < 3% de errores
Recomendaciones:
  - Investigar por que el endpoint /auth/login rechaza ~9.4% de requests
  - Verificar si el servidor tiene limitaciones de conexion simultanea
  - Revisar si hay throttling del lado del servidor (rate limiting)
  - Analizar si algunos usuarios en el CSV no tienen credenciales validas
  - Implementar reintentos automaticos para requests fallidas

Problema 2: TPS POR DEBAJO DEL OBJETIVO
Severidad: MEDIA
Evidencia: 19.90/s vs 20/s requerido (diferencia de 0.1 req/s)
Impacto: Tecnicamente NO alcanza el objetivo de 20 TPS
Recomendaciones:
  - La tasa de error reduce el throughput efectivo; resolver el Problema 1
    podria subsanar tambien este threshold
  - Revisar si hay bottlenecks en la red o en el cliente k6
  - Considerar aumentar preAllocatedVUs para absorber la demanda

CONCLUSIONES FINALES
====================

El proyecto implementa correctamente una prueba de carga con k6 siguiendo
mejores practicas (modularizacion, parametrizacion, validaciones). Sin embargo,
el servicio de destino (FakeStore API) NO CUMPLE los requisitos de SLA:

1. La tasa de error (9.39%) es mas de 3x el limite permitido (3%)
2. El throughput (19.90 TPS) queda marginalmente por debajo del objetivo (20 TPS)
3. Se registraron picos de latencia de hasta 2,633 ms en requests con error

RECOMENDACIONES PARA PROXIMAS ITERACIONES
==========================================

Corto Plazo:
  1. Ejecutar analisis de logs del servidor para entender el origen del 9.39%
     de fallas en el endpoint /auth/login
  2. Validar que los 5 usuarios del CSV tengan credenciales activas y correctas
  3. Revisar si FakeStore API aplica rate limiting o restricciones de IP

Mediano Plazo:
  1. Implementar retry logic con exponential backoff en requests fallidas
  2. Aumentar preAllocatedVUs para garantizar >= 20 TPS sostenidos
  3. Realizar tests en distintos horarios para identificar si el problema es
     de concurrencia del servidor remoto

Largo Plazo:
  1. Migrar a un endpoint interno/controlado para mediciones mas consistentes
  2. Establecer SLAs basados en el comportamiento real del servicio
  3. Implementar monitoreo continuo con alertas en tiempo real
